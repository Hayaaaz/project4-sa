1. Introduction
This report documents the storage sizing, data classification, and pgvector verification for a small school AI estate.
The goal is to estimate how much storage the estate requires, classify each asset into the correct tier, and provide proof that the pgvector extension
is installed and functioning.

The estate includes:

All sizing decisions are based on realistic assumptions and grounded in actual hardware output from my system.

2. Model Weight Sizing
Before calculating storage, I compared the two model sizes used in the course: 13B and 70B. For a small school AI assistant,

- It fits comfortably on local storage
- It is commonly used for internal RAG systems
- It provides strong performance without high cost

70B Model (INT4)

3. Vector Database Sizing
The vector store contains 5 million embeddings, each with 1536 dimensions. I used halfvec (2 bytes per dimension) because 
It is the recommended precision for pgvector, reducing storage while maintaining high recall.

Raw vectors:
5,000,000 × 1536 × 2 bytes = 15.36 GB

HNSW index overhead (~10%):
15.36 GB × 0.10 = 1.536 GB

Total vector store:
≈ 16.9 GB
-----------------------------------------------------------------------------------------------------------------------------

6. Tiering Decisions
My lsblk output shows that /dev/sda2 is rotational (ROTA=1), meaning it is a spinning HDD with low IOPS. Cold and archived data fit naturally here.
Hot‑tier assets would normally live on SSD/NVMe in production.

Asset                 Tier       Reason
---------------------------------------------------------
Model weights         Hot        Used live; requires fast access
HNSW index            Hot        Real‑time similarity search
Raw vectors           Warm       Accessed often but not per‑query
RAG corpus            Cold       Read occasionally; low IOPS acceptable
Fine‑tuning dataset   Archive    Rarely accessed; long‑term retention
Logs / scratch        Cold       Temporary metadata; low IOPS

7. Predict vs Measure Discussion
The predicted size of the empty chunks table is extremely small because it contains only metadata and page alignment.
pgvector does not store any embeddings until data is inserted. The measured size returned by PostgreSQL was 32 kB,
which matches the expected footprint for an empty table. This confirms that the pgvector extension is installed correctly 
and that the storage behavior aligns with the predicted sizing.

8. Model Choice Justification
Although the 70B model is shown for comparison, I did not choose it for this estate. It is significantly larger and more expensive to store,
  and unnecessary for a small school AI assistant.

The 13B model provides excellent performance for internal Q&A and RAG while keeping storage costs low and the estate simple to operate.
